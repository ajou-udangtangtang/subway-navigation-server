# 02. 사용자 정의 및 요구사항

## 2.1 사용자 정의

### 2.1.1 주 사용자

본 어플리케이션의 주 사용자는 **시각장애인**이며, 이하 "사용자"로 칭한다.

> 본 어플리케이션은 시각장애인(이하 '사용자')을 대상으로 하며 갤럭시 TalkBack 기능을 통한 UI 조작성을 지원해야 한다. *(원본 요구사항 명세서 中)*

### 2.1.2 사용자 특성 및 가정

| 항목 | 가정 내용 |
|---|---|
| 시각 인지 | 화면 인지 불가 또는 제한적 (전맹·저시력 모두 포함) |
| 보조 기술 | 안드로이드 **TalkBack** 사용에 익숙함 |
| 청각 | 정상 (음성 안내 인식 가능) |
| 촉각 | 정상 (진동 패턴 구분 가능) |
| 이동 능력 | 점자블록·흰 지팡이 등 보조도구를 활용하여 일정 수준 자력 이동 가능 |

### 2.1.3 비사용자

다음 그룹은 본 시스템의 직접적인 대상자가 아니다.

- 청각 장애를 동반한 시각장애인 (음성 안내가 핵심이므로)
- 휠체어 사용자 (별도의 휠체어 친화 경로 안내가 필요하므로 본 프로젝트의 범위 외)
- 시각 인지가 정상인 일반 사용자 (시각 기반 안내가 더 효율적이므로 비대상)

## 2.2 사용 환경

### 2.2.1 하드웨어 환경

| 구분 | 사양 |
|---|---|
| 대상 스마트폰 | Galaxy S24 (개발 기준 기기) |
| 운영체제 | Android (TalkBack 지원 버전) |
| 추가 하드웨어 | **없음** (스마트폰만으로 동작) |

> **변경 이력**: 4월 계획서 발표 단계에서는 ESP32 기반 BLE 비콘 + 햅틱 리모컨이 포함되어 있었으나, 이후 요구사항 재정의에 따라 **추가 하드웨어 없이 스마트폰만으로 동작**하는 구성으로 변경되었다.

### 2.2.2 사용 자세 (필수)

> 사용자의 스마트폰은 환경 정확도를 위해 **화면이 하늘을 향하게 눕혀진 상태**, **상단이 이동방향 정면을 향하도록** 한 상태를 유지한다. *(원본 요구사항 명세서 中)*

이 자세는 다음 두 기능의 정확도를 보장하기 위함이다.

- **나침반 센서**: 스마트폰 상단 방향이 곧 사용자의 이동 진행 방향이 된다.
- **Wi-Fi 신호 측정**: 일관된 자세에서 측정해야 fingerprint DB와의 패턴 일치율이 높아진다.

### 2.2.3 통신 환경

| 항목 | 내용 |
|---|---|
| 통신 방식 | HTTP POST |
| 클라이언트 라이브러리 | Retrofit (Android) |
| 서버 | Flask 웹 서버 |
| 통신 주기 | **3~5초 1회** (시각장애인의 이동 속도 및 서버 응답 지연을 고려) |

## 2.3 기능 요구사항

### 2.3.1 핵심 기능 목록

본 어플리케이션은 두 가지 핵심 기능을 제공한다.

| ID | 기능명 | 설명 |
|---|---|---|
| F-01 | 현 위치 확인 | Wi-Fi 신호를 측정해 현재 위치를 노드 단위로 추정 |
| F-02 | 이동 방향 탐색 | 다음 노드까지의 절대 방향과 사용자 진행 방향을 비교해 진동으로 안내 |

### 2.3.2 부가 기능 목록

| ID | 기능명 | 설명 |
|---|---|---|
| F-03 | 목적지 선택 | 도달 가능한 노드 목록에서 목적지를 선택 |
| F-04 | 경로 안내 | 위험 노드를 회피한 경로를 자동으로 산출 |
| F-05 | 위험 구간 안내 | 경로상 위험 구간(에스컬레이터 등) 사전 음성 안내 |
| F-06 | 잘못된 방향 감지 | 사용자가 이전 노드로 회귀했을 경우 음성으로 알림 |
| F-07 | 도착 안내 | 마지막 노드 도달 시 음성 안내 후 종료 |
| F-08 | 초기화 | 진행 상태를 초기 상태로 되돌림 |

### 2.3.3 진동 피드백 정의

방향 안내 기능 수행 시 진동 패턴은 다음과 같다.

| 사용자 진행 방향과 정답 방향의 차이 | 진동 패턴 |
|---|---|
| ±60° ~ ±30° | 초당 1회 약한 진동 |
| ±30° ~ ±15° | 초당 2회 약한 진동 |
| ±15° 이내 | (정확한 방향) — 별도 알림 없음 |

### 2.3.4 음성 피드백 정의

| 상황 | 음성 안내 문구 |
|---|---|
| 위치 도달 | "{위치}에 도달했습니다. 다음 위치의 방향을 확인하세요." |
| 방향 안내 시작 | "방향 안내를 시작합니다. 현재 위치는 {위치}입니다." |
| 위험 구간 | "{방향}방향에 보행 위험 구간이 있으므로 주의하시기 바랍니다." |
| 잘못된 방향 | "반대 방향으로 이동하였습니다." |
| 목적지 선택 불가 | "목적지로 설정 불가한 위치입니다." |
| 도착 | "목적지에 도착하였습니다." |

## 2.4 비기능 요구사항

### 2.4.1 성능 요구사항

| 항목 | 요구 수준 |
|---|---|
| API 응답 시간 | 통신 주기(3~5초)를 고려할 때, 서버 처리 시간 1초 이내 권장 |
| 위치 추정 정확도 | 노드 단위 정확도 (m 단위 정밀도는 요구되지 않음) |
| 동시 사용자 수 | 1명 (단일 사용자 시연 환경) |

> **참고**: 본 프로젝트는 **성능보다 동작 완성을 우선**으로 하므로, 위 수치는 권장 목표이며 절대 기준이 아니다.

### 2.4.2 정확도 요구사항

위치 추정은 노드 단위 정확도이며, 거리(미터 단위)에 대한 정밀도는 요구되지 않는다. 사용자가 다음 노드에 **제한 시간(Time Out) 내에 도달하는지**를 기준으로 진행 여부를 판정한다.

### 2.4.3 보안·개인정보 요구사항

- 사용자 인증·로그인 기능 **없음** (요구사항 명세서에 명시 없음).
- 측정한 Wi-Fi 데이터는 위치 추정 목적 외에는 저장·전송되지 않는다.

### 2.4.4 호환성 요구사항

- 안드로이드 **TalkBack** 보조 기능과 호환되어야 한다.
- 안드로이드 **SensorManager** API를 사용하여 자기센서(나침반) 데이터를 활용한다.
- 안드로이드 **Vibrator** API를 사용하여 진동을 출력한다.
- 안드로이드 **WiFi Scan** API를 사용하여 주변 Wi-Fi AP를 스캔한다.

### 2.4.5 데이터 품질 요구사항

> "데이터 품질 = 전체 정확도 결정"

Wi-Fi Fingerprinting 방식은 **사전 데이터 수집의 품질**이 위치 추정 정확도를 좌우한다. 따라서 각 노드별로 충분한 횟수의 신호 측정이 선행되어야 하며, 이는 팀원 전원이 공동으로 수행해야 할 과업으로 정의된다.
